# Вклад в документацию Cozystack

Этот репозиторий собирает [cozystack.ru](https://cozystack.ru) с помощью Hugo и
темы [Docsy](https://www.docsy.dev/). Здесь размещена официальная документация
Cozystack для всех поддерживаемых версий. В этом руководстве объясняется, куда
поместить ваше изменение, как работает версионированная структура и как релиз
перемещает документы.

Если вы просто хотите запустить сайт локально:

```bash
npm install
make serve    # http://localhost:1313/docs
```

См. [`README.md`](README.md) для настройки инструментов.

## Модель версионирования вкратце

Документация версионируется по **минорным релизам Cozystack**. У каждой
минорной версии есть собственный автономный каталог под `content/en/docs/`, и
каждая обслуживается под собственным префиксом URL (`/docs/v1.2/`, `/docs/v1.1/`, …).
Это позволяет пользователям более старого релиза продолжать читать точную
документацию для этого релиза, в то время как новые пользователи видят
актуальный материал.

```
content/en/docs/
├── _index.md          ← независимая от версии стартовая страница (/docs/)
├── next/              ← ствол для будущих/невыпущенных документов (/docs/next/)
├── v1.2/              ← текущая стабильная
├── v1.1/
├── v1.0/
└── v0/                ← устаревшая (все 0.x)
```

Из этой структуры следует несколько правил:

- Каждый каталог версии **независим**. Редактирование `v1.2/applications/tenant.md`
  не влияет на `v1.1/`.
- Между версиями **нет автоматического бэкпорта**. Если исправление должно
  применяться к нескольким версиям, его нужно скопировать в каждую из них.
- Автогенерируемые страницы (см. ниже) нельзя редактировать на месте — нужно
  редактировать исходный upstream-источник и заново синхронизировать.

## Почему `next/` (ствол)?

Документы, описывающие функцию, которая **ещё не вышла**, не могут находиться
ни в одном каталоге выпущенной версии:

- Они не должны попадать в `/docs/v1.2/` — это текущая продакшн-документация,
  и в v1.2 этой функции нет.
- Они не должны предварительно регистрироваться как `/docs/v1.3/` — мы пока не
  знаем, будет ли следующий релиз действительно `v1.3` и когда он выйдет.

`content/en/docs/next/` решает эту проблему: это **постоянный, независимый от
версии ствол**, где накапливаются документы для будущего релиза. Когда релиз
выпускается, содержимое `next/` продвигается в конкретный каталог версии
(`v1.3/`, `v2.0/`, …), а внутренние ссылки переписываются.

Свойства `next/`:

- **Всегда присутствует** в репозитории; никогда не удаляется.
- **Скрыт в продакшне**: исключён из сборок GitHub Pages настройкой
  `config/production/hugo.yaml` и скрыт из выпадающего списка переключателя
  версий через `hidden: true` в `hugo.yaml`.
- **Виден в локальной разработке и превью деплоя Netlify**, поэтому рецензенты
  могут увидеть документы для будущего релиза в превью PR до их выхода.
- Использует внутри пути `/docs/next/...`; они переписываются в
  `/docs/vX.Y/...` во время релиза скриптом `hack/release_next.sh`.
- На стартовой странице (`next/_index.md`) размещён **баннер-черновик**,
  предупреждающий читателя, что контент не выпущен.

## Куда должно попасть моё изменение?

Используйте эту таблицу решений перед тем, как начать редактировать:

| Ваше изменение — это…                                               | Редактируйте здесь                              |
|---------------------------------------------------------------------|--------------------------------------------------|
| Документ для **функции, которая выйдет в будущем релизе**           | `content/en/docs/next/`                         |
| **Исправление ошибки** или добавление **для текущего релиза**       | `content/en/docs/<latest_version_id>/` (`v1.2`) |
| Исправление, которое также применяется к **более старой поддерживаемой версии** | Скопируйте изменение в каждую соответствующую `vX.Y/` |
| Контент для всех версий (стартовая страница, обзор)                 | `content/en/docs/_index.md`                     |
| **Запись в блоге**                                                  | `content/en/blog/`                              |
| Изменение **макета, шорткода или общесайтового ресурса**            | `layouts/`, `assets/`, `static/`                |

Считайте `v0/` **устаревшей**: она получает исправления только для
корректности исторических ссылок. Не добавляйте туда новый контент.

## Автогенерируемый контент — не редактировать на месте

Некоторые страницы в каждой версии **синхронизируются из upstream-репозитория
[cozystack/cozystack](https://github.com/cozystack/cozystack)**. Сюда входит
большая часть `applications/`, `virtualization/`, `networking/`, `kubernetes/`
и `operations/services/`. У них есть footer:

> *Автогенерируемый контент. Не редактируйте этот файл напрямую; редактируйте источники.*

Если вам нужно изменить одну из этих страниц:

- **Изменить текст (тело)** → отредактируйте upstream README в
  `cozystack/cozystack` под `packages/apps/<app>/README.md` (или
  соответствующий `packages/extra/<svc>/`), затем повторно запустите
  соответствующую цель синхронизации: `make update-apps`, `make update-vms`,
  `make update-networking`, `make update-k8s` или `make update-services`. По
  умолчанию цели `update-*` направлены на ствол (`next/`). Передайте
  `RELEASE_TAG=vX.Y.Z`, чтобы направить в выпущенную версию для патч-обновлений.
- **Изменить только front matter** (title, weight, aliases) для конкретной
  версии → отредактируйте заглушку в
  `content/en/docs/<version>/<section>/_include/<app>.md`. Цели `update-*`
  объединяют эти заглушки с upstream README, поэтому изменения front matter
  сохраняются при следующей синхронизации. `make template-*` создаёт каркасы
  для новых заглушек.

Всё остальное на сайте написано вручную и редактируется напрямую.

## Процесс релиза

### Патч-релиз (`v1.2.5`)

`v1.2` уже существует и зарегистрирована как выпущенная версия. Workflow
релиза (upstream `cozystack/cozystack` `tags.yaml`) вызывает:

```bash
make update-all RELEASE_TAG=v1.2.5
```

Маршрутизация `RELEASE_TAG` обнаруживает, что `content/en/docs/v1.2/`
существует и не скрыт, поэтому `DOC_VERSION` разрешается в `v1.2`, и
обновление происходит **на месте** в этом каталоге. `next/` не затрагивается.

### Новый минорный или major-релиз (`v1.3.0`)

Ствол продвигается в новый каталог версии. Upstream-workflow вызывает:

```bash
make release-next RELEASE_TAG=v1.3.0
make update-all  RELEASE_TAG=v1.3.0
```

- `make release-next RELEASE_TAG=v1.3.0` вызывает `hack/release_next.sh`,
  который:
  1. Проверяет, что `next/` не пуст и что `v1.3/` ещё не существует.
  2. Копирует `content/en/docs/next/` → `content/en/docs/v1.3/` (`cp -a`).
  3. Переписывает `/docs/next/` → `/docs/v1.3/` во всех `.md`-файлах в новом
     каталоге.
  4. Обновляет `v1.3/_index.md`: устанавливает заголовок *Cozystack v1.3
     Documentation* и убирает баннер-черновик.
  5. Вызывает `hack/register_version.sh --release v1.3`, который
     регистрирует `v1.3` в `hugo.yaml`, устанавливает
     `latest_version_id: "v1.3"` и **нормализует веса `_index.md`** по всем
     не скрытым версиям (новая последняя → 10, затем 20, 30, 40, …), чтобы
     порядок в боковой панели оставался предсказуемым.
  6. Оставляет `next/` нетронутым — он остаётся на месте для следующего
     цикла релиза.

- `make update-all RELEASE_TAG=v1.3.0` затем подтягивает свежие upstream
  README, закреплённые за точным тегом, в новый каталог `v1.3/`.

#### Аудит перед релизом

`hack/release_next.sh` переписывает пути URL `/docs/next/`, но **упоминания
версий в тексте продвигаются как есть**. Перед выпуском новой минорной версии
пройдитесь по `content/en/docs/next/` на предмет любых `Cozystack v<prev>`,
`@v<prev>.0`, закреплённых версий образов или формулировок вида "since
Cozystack v<prev>", которые окажутся фактически неверными, когда ствол станет
новой версией. Шаблоны поиска, которые обычно находят такие отставшие
упоминания:

```bash
grep -rn 'Cozystack v[0-9]' content/en/docs/next/
grep -rn '@v[0-9]'           content/en/docs/next/
grep -rn 'talos:v[0-9]'      content/en/docs/next/
```

#### Сброс ствола после релиза

`release-next` **копирует** `next/` → `vX.Y/`; он не перемещает. Поэтому
ствол всё ещё содержит теперь выпущенный контент и должен быть сброшен перед
любой дальнейшей работой, направленной на следующий цикл:

```bash
make init-next
```

`init-next` очищает `content/en/docs/next/` и переинициализирует его из новой
последней версии. **Запускайте это на последнем шаге каждого релиза** —
пропуск этого шага оставляет ствол в устаревшем состоянии, когда изменения
выглядят как предназначенные для будущего релиза, но фактически накладываются
на уже выпущенный контент.

### Таблица маршрутизации

| Сценарий                                                     | Целевой каталог          |
|---------------------------------------------------------------|---------------------------|
| `make update-all` (без тега)                                  | `next/`                    |
| `make update-all RELEASE_TAG=v1.1.8` (v1.1 существует)         | `v1.1/`                    |
| `make update-all RELEASE_TAG=v1.3.0-rc1` (v1.3 ещё не существует) | `next/`                 |
| `make update-all RELEASE_TAG=v1.3.0` (v1.3 ещё не существует)  | `next/`                    |
| `make release-next RELEASE_TAG=v1.3.0`                         | копирует `next/` → `v1.3/` |
| `make update-all RELEASE_TAG=v1.3.1` (v1.3 теперь существует)  | `v1.3/`                    |

Используйте `make show-target [RELEASE_TAG=…]`, чтобы напечатать
разрешённые `DOC_VERSION` и `BRANCH` без выполнения чего-либо.

## Справочник команд

```bash
# Локальная разработка
npm install
make serve                             # dev-сервер на http://localhost:1313/docs

# Сборка в продакшн-виде (воспроизводит то, что отдаёт GitHub Pages)
./hack/download_openapi.sh && HUGO_ENV=production hugo --gc --minify

# Подтянуть upstream-контент (по умолчанию — в next/; передайте RELEASE_TAG для маршрутизации)
make update-all                        # ствол из upstream main
make update-all RELEASE_TAG=v1.2.5     # направляет в v1.2/ (патч-релиз)
make update-apps                       # только документация приложений
make update-vms | update-networking | update-k8s | update-services
make update-oss-health                 # метрики здоровья OSS (требуется Python 3.12+)

# Каркасы (заглушки для новых приложений/сервисов)
make template-apps                     # также template-vms / template-networking / …

# Жизненный цикл версии
make release-next RELEASE_TAG=v1.3.0   # продвинуть next/ → v1.3/
make init-next                         # (пере)создать next/ из последней выпущенной версии
make init-version DOC_VERSION=v1.3     # низкоуровневая инициализация любого каталога версии

# Отладка
make show-target RELEASE_TAG=v1.3.0    # печатает разрешённые DOC_VERSION / BRANCH
```

Необходимые инструменты: Hugo extended 0.160.1, Go 1.23+, Node 20+, `yq` v4+
(для целей жизненного цикла версии и маршрутизации Makefile).

## Где реализована архитектура

Если вам нужно понять или изменить сам механизм версионирования:

- `hugo.yaml` — общесайтовая конфигурация; `params.versions[]` — список
  зарегистрированных версий (включая `next` с `hidden: true`), а
  `params.latest_version_id` — текущая стабильная версия.
- `config/production/hugo.yaml` — переопределения только для продакшна;
  исключает `docs/next/**` из сборок GitHub Pages. Превью деплоя Netlify
  устанавливают `HUGO_ENV=preview`, поэтому пропускают это переопределение.
- `Makefile` — маршрутизация `RELEASE_TAG`, цели жизненного цикла.
- `hack/release_next.sh` — продвижение ствола → версии.
- `hack/register_version.sh` — управляет записями версий в `hugo.yaml` и
  нормализует веса `_index.md`.
- `hack/init_version.sh` — инициализирует любой каталог версии, с особой
  обработкой для `next/` (заголовок, баннер-черновик).
- `hack/update_apps.sh`, `hack/fill_templates.sh` — синхронизация контента из
  cozystack/cozystack.
- `hack/download_openapi.sh` — получает `openapi.json` для каждой версии во
  время сборки (v1.1+; v0 и v1.0 поставляют свой `api.json` в репозитории).
- `layouts/docs/docs-landing.html` — шаблон стартовой страницы `/docs/`;
  строит таблицу версий из `params.versions[]`, отфильтровывая скрытые
  записи.
- `layouts/partials/version-switcher.html` — выпадающий список переключателя
  версий в верхней навигационной панели; отфильтровывает скрытые записи.
- `layouts/404.html` — клиентский резервный редирект для коротких путей
  `/docs/v1/…`.

Нарушение исключительных прав влечёт ответственность в соответствии со статьёй 1301 ГК РФ (гражданско-правовая ответственность) и, в отдельных случаях, статьёй 146 Уголовного кодекса РФ (нарушение авторских и смежных прав).
