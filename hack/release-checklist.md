# Per-minor-release docs maintenance checklist

This file enumerates every spot in the website repo that must change for a new
Cozystack minor release (e.g. when `v1.3` ships or `next/` is cut to `v1.4`).
It is optimized for being fed to an agent or copilot prompt — each step has a
concrete source of truth, an exact path/key to update, and a verification
command.

> **Scope.** This checklist covers only things the website repo owns. The
> upstream release workflow in `cozystack/cozystack` (`.github/workflows/tags.yaml`)
> invokes `make update-all` / `make release-next` here; those entrypoints
> drive steps 1–3 automatically. Steps 4–7 are manual reviews.

## 0. Terminology

| Symbol         | Meaning                                                              | Example   |
| -------------- | -------------------------------------------------------------------- | --------- |
| `$RELEASE_TAG` | Full upstream tag being shipped                                      | `v1.3.0`  |
| `$DOC_VERSION` | Derived minor directory (`v<major>.<minor>`, or `v0` for 0.x)        | `v1.3`    |
| `$MINOR`       | Just the `<major>.<minor>` fragment                                  | `1.3`     |

## 1. Upstream sources of truth

Fetch these before editing the website — they drive the values in
`data/versions/<version>.yaml`:

| Pin                 | Upstream source                                                                                                    | How to read                                                                                              |
| ------------------- | ------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------- |
| `cozystack_version` | `$RELEASE_TAG` from the upstream workflow (strip leading `v`)                                                       | given as env / flag                                                                                     |
| `cozystack_tag`     | `$RELEASE_TAG` verbatim                                                                                             | given as env / flag                                                                                     |
| `talos`             | `packages/core/talos/images/talos/profiles/installer.yaml` → top-level `version:` field                             | `curl -sL https://raw.githubusercontent.com/cozystack/cozystack/$RELEASE_TAG/packages/core/talos/images/talos/profiles/installer.yaml \| yq .version` |
| `talos_minor`       | major.minor of `talos`                                                                                              | `echo $TALOS \| cut -d. -f1,2`                                                                           |

Add a new pin here (and a new key in `data/versions/*.yaml`) whenever a
release introduces a new Cozystack-release-coupled upstream version that the
docs reference literally.

## 2. Update `data/versions/$DOC_VERSION.yaml`

**One file.** The `{{< version-pin "<key>" >}}` shortcode resolves every
pinned value via this file.

- **Patch release of an existing minor** (e.g. `v1.3.1`): edit
  `data/versions/v1.3.yaml` in place — bump `cozystack_version`, `cozystack_tag`,
  and `talos` / `talos_minor` if the upstream pinned Talos changed.
- **New minor release** (e.g. cutting `v1.4.0` from `next/`):
  `hack/release_next.sh` already snapshots `data/versions/next.yaml` →
  `data/versions/v1.4.yaml` and leaves `next.yaml` alone. Just confirm the
  snapshot has the correct values, then bump `next.yaml` to point at the
  *following* target release.

**Verify:**

```bash
# Shortcode renders correctly and no key is missing
hugo --gc --minify
# Shortcode produces expected values
grep -oE 'talos:v[0-9.]+' public/docs/$DOC_VERSION/install/talos/boot-to-talos/index.html | sort -u
```

## 3. Content that must change per release (non-variable)

The `{{< version-pin >}}` shortcode covers inline version references. The
following spots are **not** covered by pins and need manual or scripted
attention each release:

### 3a. Upstream-managed app/service READMEs (automatic)

`make update-all RELEASE_TAG=$RELEASE_TAG` re-syncs `content/en/docs/$DOC_VERSION/applications/*.md`,
`virtualization/*.md`, `networking/*.md`, `kubernetes/_index.md`, and
`operations/services/*.md` from `cozystack/cozystack@$RELEASE_TAG`. It rewrites
the `source:` frontmatter to `blob/$RELEASE_TAG/...` and appends the
autogenerated notice. **Do not edit these files manually.**

### 3b. `hugo.yaml` version registry (automatic for new minors)

`hack/register_version.sh --release $DOC_VERSION` (called by `release_next.sh`)
adds `$DOC_VERSION` to `params.versions[]` and moves `params.latest_version_id`.
For patch releases, nothing changes here.

### 3c. OpenAPI spec (automatic at build time)

`hack/download_openapi.sh` fetches `openapi.json` from the release assets for
every non-hidden version. No manual step.

### 3d. Release-note blog post (manual)

Add a blog post under `content/en/blog/releases/` mirroring the upstream release
notes. Set `date` to the release date and follow the existing post structure.

## 4. Invariants to re-check after each release

Run these greps; each should return zero rows in `content/en/docs/$DOC_VERSION/`
(they catch pins that slipped in as literals):

```bash
# Inline Talos image tags that bypass the shortcode
grep -rnE 'talos:v[0-9]+\.[0-9]+\.[0-9]+' content/en/docs/$DOC_VERSION/ \
  | grep -v 'version-pin' \
  | grep -v 'v1\.11\.6'      # allow: documented boot-to-talos v0.7.1 default

# Cozystack release-asset URLs that bypass the shortcode
# (scoped to cozystack/cozystack; siderolabs/talos download URLs are unrelated)
grep -rnE 'cozystack/cozystack/releases/(download|tag)/v[0-9]+\.[0-9]+\.[0-9]+' content/en/docs/$DOC_VERSION/ \
  | grep -v 'version-pin'

# Cozystack source-tree links pinned to a literal tag (should use shortcode)
grep -rnE 'cozystack/cozystack/(blob|tree)/v[0-9]+\.[0-9]+\.[0-9]+' content/en/docs/$DOC_VERSION/ \
  | grep -v 'version-pin' \
  | grep -v '^[^:]*:[0-9]*:source: '   # allow: autogenerated frontmatter from update_apps.sh

# talos.dev version paths that bypass the shortcode
grep -rnE 'www\.talos\.dev/v[0-9]+\.[0-9]+/' content/en/docs/$DOC_VERSION/ \
  | grep -v 'version-pin'

# Leftover `latest` / `main` where a version-pin is expected (review each)
grep -rnE 'releases/latest/download|talos\.dev/latest/' content/en/docs/$DOC_VERSION/
```

Any hit needs either a shortcode retrofit (add it to the data file) or an
explicit allowlist comment above explaining why it must stay literal (e.g., the
`v1.11.6` default hardcoded in `boot-to-talos` v0.7.1).

## 5. What stays floating on `main` / `latest`

The following are intentionally **not** pinned per-release, because upstream
keeps them backward-compatible within a Cozystack minor and the UX win of
"grab the current one" outweighs determinism:

- `talm/raw/refs/heads/main/hack/install.sh` (talm installer script)
- `talm/releases/latest/download/...` (talm binaries)
- `boot-to-talos/raw/refs/heads/main/hack/install.sh`
- `boot-to-talos/releases/latest/download/...`
- `talos-bootstrap/raw/master/talos-bootstrap`
- `kubectl-etcd@latest` (from `aenix-io/etcd-operator`)

If that contract breaks (e.g., the install script grows a breaking CLI change),
promote the affected reference into `data/versions/*.yaml` as a new pin and
update this list.

## 6. Quick agent prompt

Paste this when you want an LLM to do the bump end-to-end for a patch release:

```
Cozystack $RELEASE_TAG just shipped. Update the website docs:
1. Read data/versions/$DOC_VERSION.yaml and bump cozystack_version / cozystack_tag to $RELEASE_TAG.
2. curl https://raw.githubusercontent.com/cozystack/cozystack/$RELEASE_TAG/packages/core/talos/images/talos/profiles/installer.yaml
   and bump `talos` / `talos_minor` if the upstream pinned Talos differs.
3. Run `hugo --gc --minify` and confirm no errors.
4. Run the grep invariants from hack/release-checklist.md §4.
5. Show me the diff; do not commit.
```

## 7. One-time setup for a new minor

Only needed when cutting a new minor (not for patch releases). The
`release_next.sh` + `init_version.sh` scripts already handle these; listed here
for recovery / backfill:

1. `data/versions/next.yaml` exists and reflects the next-target versions.
2. `data/versions/$DOC_VERSION.yaml` was created by `release_next.sh`
   (snapshot of `next.yaml` at cut time). If missing, run
   `cp data/versions/next.yaml data/versions/$DOC_VERSION.yaml` and edit.
3. Bump `data/versions/next.yaml` to the *following* release's target values.
